Desarrollo de Software

Especificación de requerimientos y requisitos

Proceso de entender y documentar el desarrollo de un software. El análisis de requerimientos trata de capturar y describir detalladamente los requerimientos de funcionalidad y de calidad de servicio del producto que se desarrolla.

Proceso de análisis:
• Identificar al cliente.
• Entrevistar al cliente.
• Identificar deseos y necesidades.
• Utilizar las herramientas de expresión de requisitos (las ofrecidas por UML).
• Realizar un bosquejo de las interfaces de usuario (protocolos y GUIs).
• Identificar las plataformas hardware que debe soportar el software.
• Elaborar un documento de los requisitos de usuario (debe validarse con el cliente).
• Inspeccionar los requisitos de usuario.
• Elaborar los requisitos detallados mediante documentos gráficos y textuales. Requisitos funcionales: Servicios o funciones ofrecidas. Requisitos no funcionales: Otras cualidades del sistema como performance, usabilidad, etc.

El proceso de análisis de requisitos requiere diferentes actividades que son desarrollados por múltiples agentes como usuarios, expertos de dominio, programadores, expertos de marketing, etc. Para la formulación de las especificaciones existen diferentes estándares, el más conocido es el estándar ANSI, IEEE 830-1993. Recursos para la especificación del sistema:


• Descripción del proyecto: Documento textual que describe de forma concisa el objetivo del sistema, su oportunidad de mercado y el análisis de riesgos. • Análisis del contexto: Modelo de objetos que identifica las interacciones externas y los mecanismos de interacción física entre los actores que constituyen el entorno y el propio sistema. • Casos de uso: Recursos UML para describir la funcionalidad del sistema. Identifican los límites del sistema a través de la captura de los tipos de usuario, de los elementos básicos de funcionalidad a través de casos de uso, y de los protocolos de interacción a través de diagramas de secuencia o de interacción.

Esquema del documento:
• Introducción
o Descripción general.
o Funcionalidad e interacción con otros sistemas.
o Inserción en la empresa.

• Glosario
o Definición de términos.
• Modelos del Sistema
o Diagramas de relación del sistema con el ambiente.

• Definición de requisitos funcionales
o Servicios del sistema, descripción, diagramas, prioridad (alta, baja, media).

• Definición de requisitos no funcionales (performance, usabilidad, portabilidad, etc.).
o Características del sistema, descripción, diagramas, prioridad (alta, baja, media).

• Especificación de requisitos
o Detalle de la funcionalidad: Funciones, datos, dinámica.
Validación del documento: La validación debe hacerse con el cliente o usuario a medida que se construye el documento de requisitos.
